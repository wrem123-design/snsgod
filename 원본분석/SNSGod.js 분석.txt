추가 분석 기준 파일은 RisuAI Plugin API v3용 `chat-style multi-character messenger` 플러그인입니다. 
이번에 봐야 할 핵심은 **자동화가 큐가 아니라 여러 타이머와 전역 플래그로 흩어져 있다**는 점입니다. 그래서 여러 캐릭터, 먼저 말하기, 캘린더 선톡, 자동 SNS, 이미지/프로필 갱신이 겹치면 “생성 요청이 그냥 return 되고 재시도되지 않는” 구간이 있습니다.

---

# 1. 자동화 관련 설정값 정리

## 1.1 전역 설정

원본 기본값은 다음과 같습니다.

| 설정                              |       기본값 | 실제 의미                                                  |
| ------------------------------- | --------: | ------------------------------------------------------ |
| `proactiveChatEnabled`          |   `false` | 먼저 말하기 전역 스위치                                          |
| `proactivePrivateEnabled`       |   `false` | 기본 config에는 있으나, 실행 경로에서 실제 사용되는 부분은 확인되지 않음           |
| `proactiveGroupEnabled`         |   `false` | 기본 config에는 있으나, 실행 경로에서 실제 사용되는 부분은 확인되지 않음           |
| `randomFirstMessageEnabled`     |   `false` | 랜덤 첫 메시지 타이머 사용 여부                                     |
| `randomCharacterCount`          |       `3` | 랜덤 첫 메시지를 걸 캐릭터 수, 배열 앞에서 N명                           |
| `randomMessageFrequencyMin/Max` | `10~120`분 | 랜덤 첫 메시지 지연 범위                                         |
| `autoSnsEnabled`                |   `false` | SNS 자동 생성 전역 스위치                                       |
| `autoSnsChance`                 |      `40` | 자동 SNS 확률                                              |
| `autoSnsMinMessages`            |       `6` | SNS 자동 생성 시작 전 필요한 방 메시지 수                             |
| `autoSnsCooldownMessages`       |       `4` | 마지막 SNS 이후 최소 추가 메시지 수. UI 입력은 현재 안 보이고 config 기본값만 있음 |

해당 기본값과 자동화용 상태 변수 `autonomyTimer`, `randomTimers`, `nextAutonomyAt`, `replyJobs`, `generatingRooms`, `isPosting` 등이 초반에 선언됩니다.  

## 1.2 캐릭터별 설정

캐릭터 저장 시 자동화에 직접 영향을 주는 값은 다음입니다.

| 캐릭터 필드                                 | 의미                                                    |
| -------------------------------------- | ----------------------------------------------------- |
| `enabled`                              | 캐릭터 활성 여부. 꺼져 있으면 먼저 말하기 대상에서 제외                      |
| `proactiveEnabled`                     | 캐릭터별 “먼저 말하기” 스위치                                     |
| `frequencyMinutes`                     | 먼저 말하기 후보 시간이 얼마나 자주 돌아오는지                            |
| `initiative`                           | 먼저 말하기 확률의 기본값, 0~100                                 |
| `proactiveStyle`                       | `auto`, `reserved`, `steady`, `attached`, `obsessive` |
| `proactivePatience`                    | 유저가 답장하지 않았을 때 얼마나 더 밀어붙일지                            |
| `responseDelayMin/Max`, `thinkingTime` | 실제 답장 예약/생성 지연에 영향                                    |
| `snsAutoEnabled`                       | 캐릭터별 SNS 자동 생성 허용 여부. 후반 패치에서 추가됨                     |

캐릭터 저장 함수는 `frequencyMinutes`, `initiative`, `proactiveStyle`, `proactivePatience`, `enabled`, `proactiveEnabled`를 저장하고, 저장 직후 해당 캐릭터를 다시 스케줄링한 뒤 자동화 루프를 재시작합니다. 
방/전역 설정 저장 시에는 `proactiveChatEnabled`, `randomFirstMessageEnabled`, `autoSnsEnabled`, `autoSnsChance`, `autoSnsMinMessages`를 저장하고, 랜덤 타이머와 자동화 루프를 다시 겁니다.

# 2. “먼저 말하기” 실행 순서

## 2.1 시작 시 초기화 순서

플러그인 로드 후 250ms 뒤에 다음을 실행합니다.

```ts
mgPruneSnsPosts();
mgBackfillPhoneSummaries();
mgRepairStoredPhoneMarkers();
mgForgetRoomNicknames();
saveState();

mgApplyDailyDiaryMemoryPrompt();
mgRunDailyDiaryMemoryIfDue();
mgScheduleDailyDiaryMemory();

scheduleAllCharacters(true);
scheduleRandomChats();
startAutonomyLoop();
if (mgBackgroundAutomationEnabled()) checkCalendarEvents();
```

즉, 시작 시 모든 캐릭터의 `nextAutonomyAt`을 새로 잡고, 랜덤 첫 메시지 타이머도 걸고, 5분짜리 자동화 루프를 시작합니다. 

주의할 점은 이 구간에서 `mgCalendarAutomationEnabled`는 정의되어 있는데, startup에서는 `mgBackgroundAutomationEnabled()`를 호출합니다. 파일 내에서 보이는 이름 기준으로는 서로 다릅니다. 캘린더 자동화 판정 함수는 `mgCalendarAutomationEnabled`로 정의되어 있습니다.  

## 2.2 캐릭터별 다음 발화 시간 계산

```ts
nextAt =
  now
  + frequencyMinutes * 60_000
  * random(0.65, 1.5)
  * (initial ? 0.6 : 1)
```

초기 스케줄이면 `0.65~1.5`에 다시 `0.6`을 곱하므로, 대략 `frequencyMinutes * 0.39~0.9` 사이에 첫 후보 시간이 잡힙니다. 일반 재스케줄은 `frequencyMinutes * 0.65~1.5`입니다. 

예를 들어 `frequencyMinutes = 10`이면:

```text
초기 후보: 약 3.9~9분 뒤
이후 후보: 약 6.5~15분 뒤
```

단, 실제 검사는 5분 간격 루프에서 하므로, 후보 시간이 지났다고 바로 실행되는 것은 아닙니다.

## 2.3 자동화 루프

원본 루프는 `AUTONOMY_TICK_MS = 5 * 60000`, 즉 5분마다 실행됩니다. 한 tick에서 하는 일은 다음입니다.

```ts
checkCalendarEvents();

if (!state.config.proactiveChatEnabled) return;

const due = state.characters.find(character =>
  character.enabled &&
  Date.now() >= nextAutonomyAt.get(character.id)
);

if (due) sendAutonomousMessage(due);
```

핵심은 **여러 캐릭터가 동시에 due여도 `find`로 첫 번째 캐릭터 한 명만 처리**한다는 점입니다. “가장 오래 밀린 캐릭터”가 아니라 `state.characters` 배열 순서상 가장 앞에 있으면서 due인 캐릭터입니다.

# 3. 여러 캐릭터가 있을 때 순서

## 3.1 일반 먼저 말하기 순서

캐릭터가 여러 명이면:

1. 시작 시 `scheduleAllCharacters(true)`가 모든 캐릭터에 대해 `nextAutonomyAt`을 잡습니다.
2. 5분마다 루프가 돕니다.
3. `state.characters.find(...)`로 배열 앞에서부터 due 캐릭터를 찾습니다.
4. 찾은 **한 명만** `sendAutonomousMessage`로 보냅니다.
5. 같은 tick에서 나머지 due 캐릭터는 처리하지 않습니다.

따라서 배열 순서가:

```ts
[mika, ren, yuri, haru]
```

이고 `mika`, `ren`, `yuri`가 모두 due라면, 그 tick에서는 `mika`만 시도합니다. `ren`, `yuri`는 다음 5분 tick까지 기다립니다.

## 3.2 랜덤 첫 메시지 순서

`randomFirstMessageEnabled`가 켜져 있으면 `scheduleRandomChats()`가 별도 `setTimeout`들을 만듭니다.

```ts
const enabled = state.characters
  .filter(character => character.enabled)
  .slice(0, randomCharacterCount);

for each character:
  setTimeout(() => sendAutonomousMessage(character), random(min,max) minutes)
```

여기서도 캐릭터 선택은 랜덤이 아니라 **활성 캐릭터 배열 앞에서 N명**입니다. 랜덤인 것은 발송 지연 시간뿐입니다. 그리고 이 타이머는 한 번 만들어지고, 각 타이머가 실행된 뒤 자동으로 다시 등록되지는 않습니다. 

중요한 점은 랜덤 첫 메시지도 결국 `sendAutonomousMessage(character)`를 부르기 때문에, 전역 `proactiveChatEnabled`, 캐릭터 `proactiveEnabled`, 캐릭터 `enabled` 조건을 그대로 탑니다.

# 4. 먼저 말하기 확률 계산

먼저 말하기 후보가 된 캐릭터는 바로 보내는 것이 아니라 `adjustedInitiative` 확률 검사를 통과해야 합니다.

```ts
base = character.initiative

unansweredCount = 유저 답장 이후 쌓인 proactive 메시지 수
patience = character.proactivePatience
excess = max(0, unansweredCount - patience)

style에 따라 base를 보정
최종 0~100으로 clamp
```

스타일별 공식은 다음입니다.

| style       | 공식                                                     |
| ----------- | ------------------------------------------------------ |
| `reserved`  | `initiative * max(0.08, 0.55 - excess * 0.18)`         |
| `steady`    | `initiative * max(0.2, 0.85 - excess * 0.12)`          |
| `attached`  | `initiative * max(0.45, 1.05 - excess * 0.08)`         |
| `obsessive` | `initiative * min(1.5, 1.15 + unansweredCount * 0.08)` |
| `auto`/기타   | `initiative * max(0.18, 1 - excess * 0.15)`            |

이 보정 함수는 원본 `adjustedInitiative` 그대로입니다. 

프롬프트에도 `unanswered proactive count`, `patience`, `proactiveStyle`이 들어가며, 답장이 없는 상태가 `patience`를 넘으면 “마지막 말처럼 보내거나, 그만 괴롭히겠다고 하거나, 완전히 다른 화제로 바꾸라”는 지시가 추가됩니다. 다만 코드가 강제로 중단하는 것은 아니고, 확률과 프롬프트로만 조절합니다.

# 5. 먼저 말하기가 생성되지 않는 주요 원인

## 5.1 한 tick에 한 캐릭터만 처리

현재 구조는 due 캐릭터 목록을 모두 처리하지 않고 `find` 첫 번째만 처리합니다. 그래서 여러 캐릭터가 동시에 due이면 뒤쪽 캐릭터는 최소 다음 5분 tick까지 밀립니다. 

RN에서 원본 그대로 복각하면 이런 현상이 생깁니다.

```text
12:00 tick
- A due
- B due
- C due

원본: A만 시도
B/C는 12:05 이후에야 다시 기회
```

## 5.2 busy room이 앞에 있으면 뒤 캐릭터가 막힐 수 있음

`sendAutonomousMessage`는 방이 `generatingRooms`에 있으면 바로 return 합니다.

```ts
if (!roomId || generatingRooms.has(roomId)) return;
```

그런데 이 경우 `scheduleCharacter(character, false)`를 하지 않습니다. 즉, 그 캐릭터의 `nextAutonomyAt`은 과거 시간으로 남습니다. 배열 앞쪽 캐릭터가 계속 due 상태로 남아 있으면 다음 tick에서도 다시 그 캐릭터가 먼저 잡히고, 뒤 캐릭터가 계속 밀릴 수 있습니다. 

RN에서는 이 부분을 반드시 바꿔야 합니다.

```diff
- if (generatingRooms.has(roomId)) return;
+ if (isRoomBusy(roomId)) {
+   rescheduleSoon(character, 30_000, 90_000);
+   return;
+ }
```

## 5.3 pending reply를 체크하지 않아서 유저 답장 예약을 선톡이 덮어쓸 수 있음

`sendAutonomousMessage`는 `generatingRooms.has(roomId)`만 확인합니다. 하지만 유저가 메시지를 보낸 직후, 실제 API 호출 전의 `readDelay/replyDelay` 상태에서는 `replyJobs[roomId]`와 `roomTimers[roomId]`는 있어도 `generatingRooms`는 아직 false입니다.

그 상태에서 자동화가 끼어들면 `generateCharacterReply`가 `clearRoomTimers(roomId)`를 호출하고 새 `jobId`를 넣습니다. 그러면 유저 메시지에 대한 예약 답장이 취소되고 proactive 답장으로 바뀔 수 있습니다. `generateCharacterReply`는 시작 시 기존 room timer를 지우고 새 job을 만듭니다. 

RN에서는 busy 판정을 이렇게 잡아야 합니다.

```ts
function isRoomBusy(roomId: string) {
  return (
    generatingRooms.has(roomId) ||
    !!replyJobs[roomId] ||
    !!pendingRoomStatus[roomId] ||
    (roomTimers[roomId]?.size ?? 0) > 0
  );
}
```

그리고 자동 선톡, 캘린더 선톡, 자동 SNS 후속 알림 등은 모두 이 busy check를 통과해야 합니다.

## 5.4 선톡은 첫 번째 방에만 간다

`sendAutonomousMessage`는 캐릭터의 방 목록 중 첫 번째 방만 사용합니다.

```ts
const roomId = state.chatRooms[character.id]?.[0]?.id;
```

따라서 같은 캐릭터에게 여러 방이 있어도, 먼저 말하기는 항상 첫 번째 방 기준입니다. 

RN에서 방 단위 복각을 하려면 캐릭터별 `primaryRoomId`를 명시하거나, 최근 활동 방으로 보내도록 정책을 새로 정해야 합니다.

# 6. 캘린더/기념일 선톡도 자동화 부하에 포함됨

`startAutonomyLoop`는 proactive가 꺼져 있어도 먼저 `checkCalendarEvents()`를 호출합니다. 캘린더 메시지는 “proactive first messages가 꺼져 있어도 허용”된다는 별도 프롬프트를 사용합니다.  

문제는 `checkCalendarEvents`가 이벤트를 “오늘 발송됨”으로 먼저 기록하고 저장한 뒤, `sendCalendarEventMessage`를 호출한다는 점입니다. 그리고 `sendCalendarEventMessage`는 해당 방이 generating 중이면 그냥 return 합니다.  

즉, 원본 구조에서는 이런 누락이 가능합니다.

```text
1. 오늘 기념일 이벤트 감지
2. lastTriggeredDate를 오늘로 저장
3. 방이 generatingRooms 상태라 sendCalendarEventMessage가 return
4. 메시지는 안 나감
5. 이미 오늘 발송 처리되어 재시도 안 함
```

RN에서는 캘린더/기념일도 큐에 넣고, 실제 발송 성공 후에만 `lastTriggeredDate`를 갱신해야 합니다.

```diff
- 이벤트 발견 즉시 lastTriggeredDate 저장
- 방이 busy면 return
+ 이벤트 발견 시 calendarQueue에 enqueue
+ roomBusy면 retryAt 갱신
+ deliver 성공 후 lastTriggeredDate 저장
```

# 7. SNS 자동 생성 실행 순서

## 7.1 자동 SNS는 별도 주기 루프가 아니라 “답장 완료 후” 검사

SNS 자동 생성은 주기적으로 전체 캐릭터를 훑는 구조가 아닙니다. 캐릭터 답장이 성공적으로 끝난 뒤 `runCharacterReply` 마지막에서 `maybeCreateAutoSnsPost(character, roomId)`가 호출됩니다. 

즉, SNS 자동 생성의 트리거는 다음입니다.

```text
유저 답장 / proactive 답장 / reroll 등
  -> runCharacterReply 성공
  -> deliverCharacterMessages 완료
  -> scheduleCharacter(character, false)
  -> saveState()
  -> maybeCreateAutoSnsPost(character, roomId)
```

따라서 어떤 캐릭터가 한동안 답장을 생성하지 않으면, 그 캐릭터의 SNS 자동 생성도 검사되지 않습니다.

## 7.2 자동 SNS 조건

원본 조건은 다음입니다.

```ts
if (!state.config.autoSnsEnabled) return;
if (isPosting) return;
if (!character || !roomId) return;

messages = state.messages[roomId] || [];

minMessages = max(2, autoSnsMinMessages || 6);
cooldown = max(1, autoSnsCooldownMessages || 4);

if (messages.length < minMessages) return;
if (messages.length - character.lastSnsMessageCount < cooldown) return;

chance = clamp(autoSnsChance || 40, 0, 100);
if (Math.random() * 100 >= chance) return;

isPosting = true;
try {
  await generateSnsPost(character, roomId, fixedAutoHint, { manual: false });
  await saveState();
} finally {
  isPosting = false;
  render();
}
```

이 조건은 `SNSGod.js 2849~2881`에 있습니다. 

후반 패치에서는 캐릭터별 `snsAutoEnabled === false`일 때 자동 SNS가 완전히 막힙니다. 이 토글은 SNS 앱/설정 패널 쪽에서 “이 캐릭터 SNS 자동 생성 허용”으로 표시됩니다. 

## 7.3 SNS 자동 생성은 전역 `isPosting` 하나로 막힘

`isPosting`은 캐릭터별이 아니라 전역 boolean입니다. 자동 SNS든 수동 SNS든 하나가 생성 중이면 다른 자동 SNS는 `if (isPosting) return`에서 그냥 빠집니다. 

여러 캐릭터 답장이 비슷한 시점에 끝나면 다음 상황이 생깁니다.

```text
A 답장 완료 -> maybeCreateAutoSnsPost(A) 진입 -> isPosting = true
B 답장 완료 -> maybeCreateAutoSnsPost(B) 호출 -> isPosting true라 return
C 답장 완료 -> maybeCreateAutoSnsPost(C) 호출 -> isPosting true라 return
```

B/C는 큐에 들어가지 않습니다. 다만 `lastSnsMessageCount`가 갱신되지 않았으므로, 다음에 B/C가 또 답장하면 다시 검사할 수는 있습니다. 하지만 “이번 타이밍의 자동 SNS”는 유실됩니다.

RN에서는 `isPosting`만으로 return하지 말고 SNS 생성 큐를 둬야 합니다.

# 8. SNS 생성 본체가 끝난 뒤의 후처리

자동이든 수동이든 최종 SNS 생성은 `generateSnsPost`를 탑니다. 이 함수는 캐릭터별 SNS 설정과 옵션을 합치고, LLM 호출 실패 시 fallback text를 만들고, 플랫폼 item들을 만들고, 이미지 생성까지 시도한 뒤 `state.snsPosts.unshift(post)` 합니다. 성공하면 `character.lastSnsMessageCount = messages.length`로 현재 방 메시지 수를 기록합니다. 

후반 wrapper에서 추가로:

* SNS post를 최신 30개로 prune
* hybrid면 X/Instagram item을 다양화
* daily micro mode면 이미지 자동 생성을 꺼서 짧은 일상글처럼 처리
* 작업 로그를 남김

이 붙습니다. hybrid 후처리는 `generateSnsPost` wrapper로 추가되어 있고, daily micro는 `autoImage: false`를 넣고 이미지 prompt를 비우는 방식입니다.  
작업 로그 wrapper는 `sns-auto-check`, `sns-auto-start`, `sns-auto-done`, `auto-chat-start`, `auto-chat-done` 등을 남깁니다. 단, `auto-chat-done`은 “실제로 메시지가 생성됨”이 아니라 “sendAutonomousMessage 함수 처리가 끝남”에 가깝습니다. 조기 return이어도 done 로그가 찍힐 수 있습니다.

# 9. 요청이 한꺼번에 몰릴 때 실제로 막히는 지점

## 9.1 먼저 말하기가 유저 답장 예약을 지울 수 있음

가장 위험한 케이스입니다.

```text
1. 유저가 메시지 보냄
2. generateCharacterReply가 readDelay/replyDelay timer를 걸어둠
3. 아직 runCharacterReply 전이라 generatingRooms는 false
4. 자동화 tick이 같은 캐릭터를 due로 발견
5. sendAutonomousMessage가 generatingRooms만 보고 통과
6. generateCharacterReply가 clearRoomTimers(roomId) 실행
7. 유저 답장 예약이 취소되고 proactive job으로 교체
```

원인:

```diff
- sendAutonomousMessage는 generatingRooms만 체크
+ replyJobs, pendingRoomStatus, roomTimers도 체크해야 함
```

관련 원본: `sendAutonomousMessage`의 guard와 `generateCharacterReply`의 `clearRoomTimers`.  

## 9.2 앞 캐릭터가 busy면 뒤 캐릭터가 계속 안 잡힘

```text
state.characters = [A, B, C]
A due, B due, C due

A 방이 generating 중
sendAutonomousMessage(A) return
A nextAutonomyAt은 여전히 과거
다음 tick도 A가 먼저 find됨
B/C 계속 밀림
```

원인은 `find`로 첫 due 한 명만 찾고, busy return 시 재스케줄하지 않기 때문입니다.  

## 9.3 자동 SNS는 `isPosting` 중이면 유실

자동 SNS는 `isPosting`이면 return하고 큐에 넣지 않습니다. 
수동 SNS 또는 다른 캐릭터 자동 SNS가 생성 중이면, 그 시점에 들어온 자동 SNS는 사라집니다.

## 9.4 캘린더 메시지는 busy 방에서 유실될 수 있음

캘린더 이벤트는 lastTriggered를 먼저 저장하고, 그 뒤 실제 메시지를 보내는데, 방이 busy면 발송 함수가 return합니다.  
이 경우 “오늘 발송됨”으로 저장됐지만 메시지는 없는 상태가 될 수 있습니다.

## 9.5 idle profile media update도 5분 루프에 추가됨

후반 패치에서 `startAutonomyLoop`는 원래 자동화 루프 외에 `mgIdleProfileTimer`도 5분마다 돌립니다. 이미지 생성이 켜져 있으면 모든 캐릭터를 순회하며 idle profile media update를 시도합니다. 
이 작업이 이미지/API 호출을 사용한다면, 먼저 말하기나 SNS 생성과 같은 시간대에 API 부하를 만들 수 있습니다.

# 10. RN 복각 시 권장 자동화 큐 구조

원본 동작을 기능적으로 유지하되, 유실을 막으려면 다음처럼 바꾸는 것이 좋습니다.

## 10.1 공통 room busy 판정

```ts
function isRoomBusy(roomId: string) {
  return (
    generatingRooms.has(roomId) ||
    Boolean(replyJobs[roomId]) ||
    Boolean pendingRoomStatus[roomId] ||
    ((roomTimers[roomId]?.size ?? 0) > 0)
  );
}
```

실제 TS로는:

```ts
function isRoomBusy(roomId: string): boolean {
  return (
    generatingRooms.has(roomId) ||
    !!replyJobs[roomId] ||
    !!pendingRoomStatus[roomId] ||
    ((roomTimers[roomId]?.size ?? 0) > 0)
  );
}
```

이걸 다음 모든 자동화에 적용해야 합니다.

```text
- proactive first message
- random first message
- calendar event message
- auto SNS 후속 처리
- phone-call proactive
```

## 10.2 먼저 말하기 스케줄러

원본 호환성을 살리되 밀림을 줄이려면 `find` 대신 due list를 만들어 처리하세요.

```ts
async function runAutonomyTick() {
  enqueueDueCalendarEvents();

  if (!config.proactiveChatEnabled) return;

  const dueCharacters = characters
    .filter(c => c.enabled !== false)
    .filter(c => Date.now() >= (nextAutonomyAt.get(c.id) ?? 0))
    .sort((a, b) => {
      const atA = nextAutonomyAt.get(a.id) ?? 0;
      const atB = nextAutonomyAt.get(b.id) ?? 0;
      return atA - atB;
    });

  for (const character of dueCharacters) {
    enqueueProactiveJob(character);
  }

  drainAutomationQueue();
}
```

원본과 달라지는 점은 “한 tick에 한 명만”이 아니라 due된 캐릭터를 큐에 모두 넣는 것입니다. 실제 실행은 API 부하를 막기 위해 concurrency 1로 두는 게 안전합니다.

## 10.3 proactive job 실행

```ts
async function runProactiveJob(character: Character) {
  if (!config.proactiveChatEnabled) return;
  if (character.enabled === false) return;
  if (character.proactiveEnabled === false) return;

  const roomId = getPrimaryRoomId(character.id);
  if (!roomId) return;

  if (isRoomBusy(roomId)) {
    rescheduleSoon(character, 30_000, 90_000);
    return;
  }

  const unanswered = unansweredProactiveCount(roomId);
  const chance = adjustedInitiative(character, unanswered);

  if (Math.random() * 100 > chance) {
    scheduleCharacter(character, false);
    return;
  }

  await generateCharacterReply(character, roomId, '', 'proactive');
}
```

원본과 비교한 diff:

```diff
- if (!roomId || generatingRooms.has(roomId)) return;
+ if (!roomId) return;
+ if (isRoomBusy(roomId)) {
+   rescheduleSoon(character);
+   return;
+ }

- 한 tick에 state.characters.find(...) 한 명만 실행
+ due list 전체를 큐에 넣고 concurrency 제한으로 처리
```

## 10.4 캘린더 job 실행

원본은 이벤트를 먼저 triggered 처리하는데, RN에서는 발송 성공 후 처리하세요.

```ts
async function runCalendarJob(character, event, today) {
  const roomId = getPrimaryRoomId(character.id);
  if (!roomId) return;

  if (isRoomBusy(roomId)) {
    retryCalendarJobLater(character, event, today);
    return;
  }

  await sendCalendarEventMessage(character, event, today);

  markCalendarEventTriggered(character, event, today);
  await persist();
}
```

diff:

```diff
- lastTriggeredDate를 send 전 저장
+ deliver 성공 후 저장

- generatingRooms면 return
+ busy면 retry 큐로 이동
```

## 10.5 SNS 자동 생성 큐

`isPosting` 때문에 자동 SNS가 유실되는 구조를 큐로 바꾸세요.

```ts
type AutoSnsJob = {
  key: string; // auto-sns:${characterId}:${roomId}
  characterId: string;
  roomId: string;
  enqueuedAt: number;
  sourceMessageCount: number;
};

const snsQueue: AutoSnsJob[] = [];
let snsPosting = false;

function enqueueAutoSnsCheck(character, roomId) {
  const key = `auto-sns:${character.id}:${roomId}`;

  const existing = snsQueue.find(j => j.key === key);
  const count = state.messages[roomId]?.length ?? 0;

  if (existing) {
    existing.sourceMessageCount = Math.max(existing.sourceMessageCount, count);
    return;
  }

  snsQueue.push({
    key,
    characterId: character.id,
    roomId,
    enqueuedAt: Date.now(),
    sourceMessageCount: count,
  });

  drainSnsQueue();
}
```

실행 시점에 원본 조건을 다시 검사합니다.

```ts
async function runAutoSnsJob(job: AutoSnsJob) {
  const character = getCharacter(job.characterId);
  if (!character) return;

  if (character.snsAutoEnabled === false) return;
  if (!config.autoSnsEnabled) return;

  const messages = state.messages[job.roomId] ?? [];
  const minMessages = Math.max(2, config.autoSnsMinMessages || 6);
  const cooldown = Math.max(1, config.autoSnsCooldownMessages || 4);

  if (messages.length < minMessages) return;
  if (messages.length - (character.lastSnsMessageCount || 0) < cooldown) return;

  const chance = clamp(config.autoSnsChance || 40, 0, 100);
  if (Math.random() * 100 >= chance) return;

  await generateSnsPostFinal(character, job.roomId, AUTO_SNS_HINT, {
    manual: false,
  });
}
```

drain은 직렬로:

```ts
async function drainSnsQueue() {
  if (snsPosting) return;
  snsPosting = true;

  try {
    while (snsQueue.length) {
      const job = snsQueue.shift()!;
      await runAutoSnsJob(job);
    }
  } finally {
    snsPosting = false;
  }
}
```

원본 대비 diff:

```diff
- if (isPosting) return;
+ enqueueAutoSnsCheck(...)

- auto SNS 검사 도중 다른 auto SNS는 유실
+ serial queue에서 순서대로 재검사

- character.lastSnsMessageCount는 생성 성공 시만 갱신
+ 동일하게 유지
```

# 11. 원본 동작을 최대한 보존한 최종 순서도

## 11.1 유저 메시지 후

```text
User sends message
  -> sendUserMessage
  -> generateCharacterReply(mode='reply')
  -> runCharacterReply
  -> deliver messages
  -> scheduleCharacter(character, false)
  -> maybeCreateAutoSnsPost
       -> global auto SNS on?
       -> character auto SNS on?
       -> min/cooldown/chance ok?
       -> SNS queue or generateSnsPost
```

## 11.2 먼저 말하기 루프

```text
startup
  -> scheduleAllCharacters(true)
  -> scheduleRandomChats()
  -> startAutonomyLoop()

every 5 minutes
  -> checkCalendarEvents()
  -> if proactiveChatEnabled:
       dueCharacters = original: first due only
       recommended RN: all due sorted by nextAt
       -> sendAutonomousMessage
            -> global proactive on?
            -> character enabled?
            -> character proactiveEnabled?
            -> primary room exists?
            -> room not busy?
            -> adjustedInitiative chance pass?
            -> generateCharacterReply(mode='proactive')
```

## 11.3 랜덤 첫 메시지

```text
save/startup
  -> scheduleRandomChats()
      if randomFirstMessageEnabled:
        enabled characters.slice(0, randomCharacterCount)
        setTimeout(random min~max minutes)
        -> sendAutonomousMessage(character)
```

즉, 랜덤 첫 메시지는 “랜덤 캐릭터 선택”이 아니라 “앞에서 N명에게 랜덤 지연”입니다. 

## 11.4 SNS 자동 생성

```text
runCharacterReply success
  -> maybeCreateAutoSnsPost(character, roomId)

maybeCreateAutoSnsPost
  -> global autoSnsEnabled?
  -> not isPosting?
  -> character exists?
  -> room exists?
  -> character.snsAutoEnabled !== false?
  -> messages.length >= autoSnsMinMessages?
  -> messages.length - lastSnsMessageCount >= autoSnsCooldownMessages?
  -> random chance pass?
  -> generateSnsPost(manual=false)
  -> lastSnsMessageCount = current room message count
```

# 12. RN 구현 체크리스트

## 필수 수정

```diff
+ 자동화 큐를 만든다.
+ roomBusy는 generatingRooms뿐 아니라 replyJobs, pendingRoomStatus, roomTimers까지 본다.
+ due 캐릭터는 find 1명이 아니라 due list 전체를 큐에 넣는다.
+ busy로 실패한 proactive/calendar는 짧은 retry로 재스케줄한다.
+ calendar lastTriggered는 실제 발송 성공 후 저장한다.
+ SNS 자동 생성은 isPosting이면 return하지 말고 queue에 넣는다.
+ auto-chat-done 로그를 “메시지 생성 완료”로 해석하지 않는다.
```

## 원본과 동일하게 유지할 부분

```diff
+ proactive 확률 공식은 adjustedInitiative 그대로 사용
+ unansweredProactiveCount는 마지막 user message 이후 proactive sourceMode만 count
+ scheduleCharacter 랜덤 배율은 0.65~1.5, initial이면 *0.6
+ SNS 자동 조건은 minMessages, cooldownMessages, chance, character.snsAutoEnabled 유지
+ SNS 생성 성공 시 character.lastSnsMessageCount 갱신
+ SNS 생성 실패 시 가능한 fallback post를 만들고 앱이 멈추지 않게 처리
```

# 13. 가장 의심되는 현재 버그 원인

지금 “요청이 한꺼번에 몰려서 생성을 못하는 경우”는 소스 기준으로 아래 가능성이 큽니다.

1. **먼저 말하기가 유저 답장 예약을 취소**
   `sendAutonomousMessage`가 `replyJobs/roomTimers`를 안 보고 들어와서 `generateCharacterReply`의 `clearRoomTimers`가 기존 답장 예약을 지울 수 있습니다.

2. **앞쪽 캐릭터 busy 때문에 뒤쪽 캐릭터가 계속 밀림**
   `state.characters.find`로 한 명만 고르고, busy return 시 재스케줄이 없습니다.

3. **자동 SNS가 `isPosting` 중이면 유실**
   전역 `isPosting`이 true면 자동 SNS는 큐 없이 return합니다.

4. **캘린더 메시지가 busy 상태에서 오늘 발송 처리만 되고 메시지는 없음**
   `lastTriggeredDate`를 먼저 저장한 뒤 room busy면 발송 함수가 return합니다.

5. **랜덤 첫 메시지와 일반 먼저 말하기가 같은 `sendAutonomousMessage`를 공유**
   랜덤 첫 메시지도 전역 proactive/캐릭터 proactive/busy 조건에 걸려 조용히 return할 수 있습니다.

6. **idle profile media update가 5분마다 모든 캐릭터를 순회**
   이미지 생성이 켜져 있으면 자동화 루프와 같은 주기로 추가 API/이미지 부하가 생길 수 있습니다.

RN에서는 위 6개를 큐/락/재시도 구조로 바꾸면, 원본 기능은 유지하면서 “한꺼번에 몰려서 생성이 사라지는” 문제를 크게 줄일 수 있습니다.


아래는 **React Native 복각용 기능 명세 + diff 수준 구현 가이드**입니다. 원본은 RisuAI Plugin API v3 위에서 도는 웹 플러그인이고, 파일 자체도 `chat-style multi-character messenger` 플러그인으로 시작합니다. 

핵심 결론부터 말하면, 이 파일은 **초반 함수가 최종 함수가 아닙니다.** `buildMessages`, `deliverCharacterMessages`, `buildSnsPostMessages`, `generateSnsPost`, `maybeCreateAutoSnsPost`가 뒤쪽에서 계속 래핑/재정의됩니다. RN에서 초반 원본 함수만 베끼면 채팅 출력과 SNS 생성이 반드시 꼬입니다. 구현은 “원본 함수 복사”가 아니라 **최종 파이프라인 재구성**으로 해야 합니다.

# 2. 채팅 답장 출력 파이프라인

## 2.1 최종 채팅 흐름

RN에서는 아래 순서를 그대로 구현해야 합니다.

```ts
sendUserMessage(text, attachment)
  -> cancelPendingReply(roomId)
  -> push user attachment message first
  -> push user text message second
  -> save room + render
  -> generateCharacterReply(character, roomId, latestUserText, "reply")

generateCharacterReply(...)
  -> if room already generating: create retry job and retry after 700ms
  -> clear old room timers
  -> create jobId
  -> compute readDelay/replyDelay
  -> pendingRoomStatus = "sent" or "read"
  -> after readDelay: status = "read"
  -> after replyDelay: runCharacterReply(...)

runCharacterReply(...)
  -> stale job guard
  -> generatingRooms.add(roomId)
  -> pendingRoomStatus = "thinking"
  -> payload = callLLM(await buildMessages(...))
  -> stale job guard again
  -> parse/normalize payload
  -> deliverCharacterMessages(...)
  -> update room.lastActivity
  -> save
  -> maybeCreateAutoSnsPost(character, roomId)
  -> finally clear job/status/generatingRooms
```

원본 `sendUserMessage`는 첨부파일이 있으면 먼저 media message를 넣고, 텍스트가 있으면 그 다음 user text message를 넣은 뒤 `generateCharacterReply`를 호출합니다. 
`generateCharacterReply`는 방 단위 jobId, read delay, reply delay, pending status를 만들고, 이미 생성 중인 방이면 최신 요청으로 다시 예약합니다. 
실제 LLM 호출은 `runCharacterReply`에서 이뤄지며, 응답 후 normalize → deliver → save → auto SNS 체크까지 이어집니다. 

### RN diff 지시

```diff
- sendMessage() 안에서 바로 await api.reply() 후 setMessages([...])
+ 방별 replyJobId, generatingRooms, roomTimers를 둔다.
+ 사용자가 새 메시지를 보내면 기존 pending reply/timer를 반드시 취소한다.
+ API 응답 직후에도 jobId가 최신인지 확인한다.
+ 이미지 생성/말풍선 gap 대기 후에도 jobId를 다시 확인한다.
```

## 2.5 LLM 호출 규칙

### 채팅과 SNS의 차이


## 2.6 채팅 응답 파싱/정규화

원본 기본 parser는 JSON object를 꺼내고, 실패하면 `content/body/text/imageCaption` 같은 필드를 regex로 추출해 fallback message를 만듭니다. 
또 JSONish repair와 sanitize 과정에서 코드펜스, JSON scaffold, `content:` 같은 메타 누수를 제거합니다. 

뒤쪽에서는 `cleanChatOutputText`가 추가되어 JSON 메타 누수, `Final Response`, code fence 등을 더 강하게 제거하고, 화면 렌더링 직전에도 한 번 더 정리합니다. 

RN 정규화는 다음처럼 구현하세요.

```ts
type NormalizedReplyItem = {
  delay: number;
  content: string;
  sticker?: string;
  imagePrompt?: string;
  imageCaption?: string;
  giftType?: string;
  giftMessage?: string;
  callInvite?: boolean;
  callTitle?: string;
  callLine?: string;
};
```

```ts
function normalizeReplyMessages(character, payload): NormalizedReplyItem[] {
  let items = Array.isArray(payload.messages) ? payload.messages : [];

  items = items.map(item => ({
    delay: number(item.delay),
    content: cleanChatOutputText(item.content ?? item.body ?? item.text ?? ''),
    sticker: stringOrEmpty(item.sticker),
    imagePrompt: trim(item.imagePrompt),
    imageCaption: cleanChatOutputText(item.imageCaption ?? ''),
    ...normalizeGiftItem(item),
    ...normalizePhoneItem(item),
  })).filter(hasVisibleOrSpecialContent);

  if (character.messageStyle === 'long' && items.length > 1 && !items.some(i => i.giftType)) {
    return mergeIntoOneLongBubble(items);
  }

  return addPhoneMarkerForExplicitRequest(character, payload, items);
}
```

주의: 초기 `normalizeReplyMessages`에는 `burst` 자동 분할이 있었지만, 뒤쪽 gift 패치에서 함수가 통째로 재정의되면서 최종 함수에는 그 분할 로직이 남아 있지 않습니다. 초기 코드만 보고 `burst`를 또 자동분할하면 원본 최종 동작과 달라질 수 있습니다. 초기 style 처리와 image instruction은 여기서 확인됩니다.

## 2.7 메시지 출력 순서

최종 deliver는 개념상 아래 순서입니다.

```ts
async function deliverCharacterMessages(character, roomId, items, jobId, sourceMode) {
  items = expandPhoneMarkersToCallInvites(items);
  items = maybeUpgradePhoneIntent(items);
  items = stripPhoneInvitesIfDisabled(items);

  for (let i = 0; i < items.length; i++) {
    if (!isCurrentJob(roomId, jobId)) return;

    if (i > 0) await delay(messageGapSeconds(character, items[i]));

    if (!isCurrentJob(roomId, jobId)) return;

    if (hasDirectPhoneCall(items[i])) {
      pushPhoneCallCard(...);
      continue;
    }

    if (items[i].giftType) pushGiftMessage(...);
    if (items[i].sticker) pushStickerMessage(...);
    if (items[i].imagePrompt && config.imageGeneration.enabled) {
      tryPushGeneratedImage(...);
    }
    if (items[i].content || items[i].imageCaption && !items[i].giftType) {
      pushTextMessage(...);
    }
  }
}
```

전화 관련 final wrapper는 `callInvite`, `phoneCall`, `callTitle`, `callLine`, 또는 `[[PHONE_CALL]]`을 phone-call card로 바꿉니다. 
전화 마커/초대 판정은 최종적으로 `[[PHONE_CALL]]` 패턴과 `callInvite === true` 중심으로 정리됩니다. 

### RN에서 흔한 버그

```diff
- API 응답이 늦게 도착하면 그대로 append
+ jobId가 현재 replyJobs[roomId]와 다르면 버린다.

- 이미지 생성 완료 후 바로 append
+ 이미지 생성 후에도 jobId stale check를 한다.

- content만 렌더링
+ mediaType image/gift/phone-call, sticker를 별도 bubble/card로 렌더링한다.

- callTitle/callLine이 있으면 바로 전화 생성
+ 최종 규칙은 callInvite true 또는 PHONE_CALL marker가 기준이다.
  단, 원본 direct phone wrapper는 callTitle/callLine 단독도 phone item으로 흡수하므로
  RN에서는 normalize 단계에서 callInvite true로 승격하거나, delivery에서 direct phone으로 처리한다.
```

## 3.4 최종 SNS prompt builder

초기 `buildSnsPostMessages`는 Lightboard JSON schema, 댓글, DM, 플랫폼 옵션을 이미 포함합니다. 
하지만 최종 버전은 `mgSnsBuildCoreMessages`입니다. 이 함수는 깨끗한 DM timeline, phone summary, direction, mood, attachment 여부, previous posts, lore, tagging guide를 합쳐 **SNS 전용 JSON**을 요구합니다. 

최종 builder 핵심:

```ts
function buildSnsMessages(character, hintText, roomId, options) {
  const sns = merge(DEFAULT_CONFIG.sns, state.config.sns, options.sns);
  const direction = mgSnsEffectiveDirection(hintText, sns);
  const transcript = mgSnsRecentCleanMessages(roomId, character, mgSnsContextMessageLimit()).join('\n');
  const phoneSummary = mgRecentPhoneContextForRoom(roomId, character, 2);
  const platform = sns.platform === 'hybrid' ? 'twitter and instagram' : sns.platform;

  const system = [
    snsPostingPrompt,
    'Create a Lightboard SNS post result as one valid JSON object.',
    'Schema: {"platforms":[...],"dms":[...]}',
    `Target platform: ${platform}`,
    `Visible language: ${config.language}`,
    `Character: ${character.name} (@${character.handle})`,
    `User: ${userName}. Profile: ${userProfile}`,
    `Character profile: ${character.prompt}`,
    lorePromptBlock(...),
    characterTaggingGuide(character),
    memories,
    writingRulesFromOptions(sns),
  ].join('\n');

  const user = [
    `Clean recent private DM timeline with ${character.name}:`,
    transcript || '(no recent chat text; use the character profile and direction)',
    'Recent phone-call summaries:',
    phoneSummary || '(none)',
    `User direction for this post: ${direction || '(none)'}`,
    `Mood: ${clean(sns.mood) || '(none)'}`,
    `Attached image: ${options.image ? 'yes' : 'none'}`,
    'Create the SNS JSON now.',
  ].join('\n');

  return [{ role: 'system', content: system }, { role: 'user', content: user }];
}
```

### 옵션별 동작

| 옵션                      | 원본 동작                                    | RN 구현                            |
| ----------------------- | ---------------------------------------- | -------------------------------- |
| `platform: 'twitter'`   | platform item 1개                         | `platforms = ['twitter']`        |
| `platform: 'instagram'` | platform item 1개                         | `platforms = ['instagram']`      |

## 3.6 SNS platform normalize

최종 `sanitizeSnsPlatform`은 다음을 보장합니다.

* platform 값 보정
* displayName/handle 기본값
* text 정제
* phone artifact 제거
* hashtags 보정
* stats 기본값 생성
* comments 수 보정
* image/imagePrompt/imageCaption 정제

해당 로직은 `SNSGod.js 20705~20884`에서 확인됩니다. 

RN 구현 예:

```ts
function normalizeSnsPlatforms(platforms, sns, character, postContext) {
  const expected =
    sns.platform === 'hybrid'
      ? ['twitter', 'instagram']
      : [sns.platform === 'instagram' ? 'instagram' : 'twitter'];

  const source = platforms?.length
    ? [...platforms]
    : expected.map(platform => ({ platform, text: '' }));

  if (sns.platform === 'hybrid') {
    for (const p of expected) {
      if (!source.some(item => item.platform === p)) {
        source.push({ platform: p, text: source[0]?.text ?? '' });
      }
    }
  }

  return source
    .slice(0, expected.length)
    .map((item, index) => sanitizeSnsPlatform({
      ...item,
      platform: isKnownPlatform(item.platform) ? item.platform : expected[index],
      __sns: sns,
    }, postContext, character))
    .filter(item => item.text || item.image || item.imagePrompt);
}
```

## 3.7 이미지 생성

최종 SNS 생성에서 플랫폼 item마다 다음 순서가 적용됩니다.

```ts
for each platform item:
```

## 3.9 SNS 후반 wrapper까지 반영해야 하는 부분

최종 SNS 동작은 `generateSnsPost` 본체 이후에도 래퍼가 붙습니다.

| wrapper                          |         원본 위치 | 기능                                            |
| -------------------------------- | ------------: | --------------------------------------------- |
| post retention                   |    `21178` 부근 | SNS post 최신 30개 유지                            |
| hybrid dedupe/diversify          | `22374~22503` | hybrid면 X/Instagram item을 다양화, nsfw split 처리  |
| fresh comments + third-party DMs | `22580~22664` | 댓글 중복 방지, DM 없으면 follower DM 생성               |
| daily micro post                 | `28672~28741` | 일상 짧은 글 모드면 autoImage false, imagePrompt 제거   |
| operation log                    | `30576~30607` | 수동/자동 시작/완료/실패 로그                             |

RN에서 이걸 “함수 재정의”로 따라 할 필요는 없습니다. 아래처럼 명시적 pipeline hook으로 만들면 됩니다.

```ts
async function generateSnsPostFinal(character, roomId, hintText, options) {
  logStart();

  const sns = resolveSnsConfig(character, options);
  const daily = detectDailyMicroMode(character, roomId, hintText, options, sns);
  const adjustedOptions = applyDailyMicroOptions(options, daily);

  let post = await generateSnsPostCore(character, roomId, hintText, adjustedOptions);

  pruneSnsPosts(30);

  if (sns.platform === 'hybrid') {
    await diversifyHybridSnsPost(post, character, sns, roomId, hintText, options);
  }

  freshenSnsPostComments(post, character, sns);
  ensurePostThirdPartyDms(post, character, sns);

  if (daily) stripDailyMicroImages(post);

  logDone();
  return post;
}
```

# 4. 전화/통화 로그가 SNS를 오염시키는 문제

원본에서 SNS 꼬임을 막으려고 가장 많이 패치된 부분이 **전화 로그 제거**입니다. RN에서도 꼭 분리해야 합니다.

원본은 SNS timeline line 생성 시:

* `mediaType === 'phone-call'` 제외
* `phoneLog` 제외
* call prelude text 제외
* `PHONE_CALL`, 통화기록, 부재중, 취소 같은 UI artifact 제거
* 그래도 최근 통화가 의미 있으면 “전화했다” 정도의 자연스러운 summary만 사용

관련 정제 함수는 `mgSnsCleanTimelineText`, `mgSnsRecentCleanMessages` 쪽입니다. 
뒤쪽에서는 call prelude까지 걸러서 SNS transcript에 넣지 않습니다. 

```diff
- SNS prompt에 최근 메시지를 그대로 join
+ SNS 전용 clean timeline을 만든다.

- phone-call card, 통화기록, [[PHONE_CALL]]를 SNS text/comment/hashtag/imagePrompt에 넣음
+ phone summary만 자연어 context로 넣고 UI label은 제거
```

# 7. 구현 우선순위

1. **ChatEngine job/timer/stale guard부터 복구**
   채팅 출력 에러 대부분은 중복 요청, stale 응답, 이미지 생성 후 stale append에서 납니다.

2. **OutputParser와 MessageDelivery 분리**
   `content`만 append하지 말고 `gift`, `phone-call`, `image`, `sticker`를 message model로 분해하세요.

3. **최종 `buildMessages` 기준으로 prompt 재구성**
   `SNSGod.js 3498~3567` 이후의 builder를 기준으로 하고, 뒤쪽 wrapper는 middleware로 추가하는 구조가 안전합니다.

4. **SNS는 `platforms[]` 모델로 다시 잡기**
   `post.text` 중심 구현이면 hybrid, Instagram, comments, DMs가 계속 꼬입니다.

5. **SNS 전용 timeline cleaner 구현**
   채팅 transcript cleaner와 SNS transcript cleaner를 분리해야 통화/전화 UI artifact가 SNS에 새지 않습니다.

6. **SNS final pipeline hook 구현**
   core generate 후 `retention → hybrid diversify → fresh comments → DMs → daily micro cleanup → ops log` 순서로 후처리하세요.

```ts
async function runCharacterReply(character, roomId, userText, mode, replyDelay, jobId) {
  if (!isCurrentJob(roomId, jobId) || generatingRooms.has(roomId)) return;

  generatingRooms.add(roomId);
  setPending(roomId, 'thinking', jobId);

  try {
    const llmMessages = await buildChatMessagesFinal(character, { roomId, userText, mode, replyDelay });
    const payload = await llmClient.call(llmMessages);
    if (!isCurrentJob(roomId, jobId)) return;

    const items = normalizeReplyMessagesFinal(character, { ...payload, userText, latestUserText: userText });
    await deliverCharacterMessagesFinal(character, roomId, items, jobId, mode);

    updateRoomActivity(roomId);
    await persist();
    await maybeCreateAutoSnsPost(character, roomId);
  } catch (e) {
    pushSystemMessage(roomId, `전송 실패: ${e.message}`);
    await persist();
  } finally {
    generatingRooms.delete(roomId);
    clearJobIfCurrent(roomId, jobId);
  }
}
```

```ts
// SNS
async function generateSnsPostFinal(character, roomId, hintText, options) {
  logSnsStart(character, roomId, options);

  const sns = resolveSnsConfig(character, options);
  const daily = detectDailyMicroMode(character, roomId, hintText, options, sns);
  const nextOptions = daily
    ? { ...options, __snsDailyMicroMode: true, sns: { ...options.sns, autoImage: false } }
    : { ...options, __snsDailyMicroMode: false };

  let post = await generateSnsPostCore(character, roomId, hintText, nextOptions);

  pruneSnsPosts(30);

  if (sns.platform === 'hybrid') {
    await diversifyHybridSnsPost(post, character, sns, roomId, hintText, options);
  }

  freshenSnsPostComments(post, character, sns);
  ensurePostThirdPartyDms(post, character, sns);

  if (daily && !options.image) stripPostImagesIfMissing(post);

  logSnsDone(post);
  return post;
}

async function generateSnsPostCore(character, roomId, hintText, options) {
  const sns = sanitizeSnsConfig(resolveSnsConfig(character, options), hintText);
  const direction = effectiveSnsDirection(hintText, sns);

  let raw = '';
  let fallbackReason = '';

  try {
    const messages = buildSnsMessagesFinal(character, direction, roomId, { ...options, sns });
    raw = await llmClient.callRaw(messages, {
      maxTokens: Math.max(profile.maxTokens, sns.platform === 'hybrid' ? 3600 : 2800),
      responseMimeType: false,
    });
  } catch (e) {
    fallbackReason = e.message;
    raw = freshFallbackSnsText(character, roomId, direction);
  }

  const parsed = parseSnsPayloadFromRaw(raw, sns, character, roomId, direction);
  const platforms = ensureSnsPlatforms(parsed, sns, character, roomId, direction);

  await attachOrGenerateSnsImages(platforms, character, sns, options.image, raw, direction);

  const first = platforms[0] ?? {};
  const post = makeSnsPost({ character, roomId, direction, platforms, parsed, first, raw, fallbackReason, manual: options.manual });

  state.snsPosts.unshift(post);
  character.lastSnsMessageCount = state.messages[roomId]?.length ?? 0;
  notifySnsPost(character, post);

  return post;
}
```
