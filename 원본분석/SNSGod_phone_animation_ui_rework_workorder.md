# SNSGod 통화 기능 원본형 UI/애니메이션 복각 작업지시서

대상: 현재 React Native 구현의 `CallScreen` / `PhoneCallScreen` 계열 통화 UI  
목표: 현재 구현 상태(정선 스크린샷 3장)를 **원본 SNSGod 통화 연출**에 최대한 가깝게 수정한다.

이 문서는 단순 아이디어 메모가 아니라, **현재 구현과 원본의 차이 → 반드시 수정해야 하는 UX/애니메이션 → 상태 머신 → 컴포넌트 구조 → 렌더링 규칙 → 타이핑/페이지 넘김/통화 링 애니메이션 상세 → 완료 기준**까지 포함한 **실전 작업지시서**다.

---

## 0. 결론 요약

현재 구현은 이미 예전보다 나아졌지만, 여전히 원본과 가장 크게 다른 핵심은 아래 6개다.

1. **캐릭터 발화를 “채팅 transcript”처럼 계속 누적해서 보여준다.**  
   원본은 기본적으로 **한 번에 한 박스/한 턴 중심**이다. 여러 줄이 한 카드에 들어가더라도, 체감은 “지금 이 한 마디를 듣는 중”에 가깝다.

2. **한 문장/한 턴이 박스 안에 한 글자씩 생성되는 typewriter 연출이 부족하다.**  
   원본은 사각 박스 안에서 대사가 **실시간으로 입력되는 느낌**이 강하다.

3. **프로필 사진 주변의 통화 링/펄스 애니메이션이 약하다.**  
   원본은 전화 연결 중/대화 중/듣는 중 상태가 프로필 주변의 링 연출로 더 직관적으로 보인다.

4. **상태 문구와 행동 흐름이 원본처럼 분명하지 않다.**  
   원본은 `연결 중...`, `통화 연결됨`, `듣는 중...`, `내 응답을 기다리는 중`, `통화 종료 중` 같은 상태가 단순 텍스트가 아니라 **현재 턴 흐름을 설명하는 UX 일부**다.

5. **한마디마다 다음 페이지/다음 턴으로 넘기는 구조가 약하다.**  
   원본은 긴 통화 transcript 스크롤 UI보다, **현재 발화 카드 → 다음 버튼/선택지 → 다음 발화 카드**처럼 진행되는 연출이 중심이다.

6. **선택지/직접답하기/말풍선/상태박스의 레이아웃 우선순위가 원본과 다르다.**  
   현재 구현은 “통화 + 메신저 혼합형”이고, 원본은 “통화용 시네마틱 카드 UI”에 더 가깝다.

즉, 이번 작업은 단순 스타일 수정이 아니라,  
**“통화 transcript형 UI”를 “원본형 턴 진행식 통화 UI”로 바꾸는 작업**이다.

## 1. 현재 구현 vs 원본 차이 정리

## 1-1. 현재 구현(정선 스크린샷 3장) 특징

현재 구현은 대략 이런 흐름이다.

- 상단: `SNSGod 통화`
- 중앙: 프로필 이미지 / 이름 / 상태 문구 / 연결 시간
- 중단: 캐릭터/유저 말풍선이 누적됨
- 하단: 선택지 버튼 3개 또는 직접 입력
- 최하단: `끊기`

문제는 이 구조가 **채팅 앱 느낌**을 너무 남긴다는 점이다.

### 현재 구현 문제점

1. 발화가 여러 개 누적되며, “지금 이 한마디를 듣는 중” 느낌이 약함.
2. 캐릭터가 말하는 도중에도 화면이 transcript 쌓기처럼 보여서 원본의 시네마틱함이 줄어듦.
3. `정선이 말하는 중...`이 텍스트로는 보이지만, 실제 애니메이션/카드 흐름은 그 상태를 충분히 전달하지 못함.
4. 하단에서 선택지/입력창/상태 라벨이 동시에 많아 보여 약간 복잡함.
5. 직접 답하기가 원본 대비 존재감이 너무 큼.
6. 캐릭터 발화가 박스 안에서 “문장 생성”이라기보다, 이미 렌더된 메시지가 들어온 것처럼 보일 수 있음.

---

## 1-2. 원본(나머지 스크린샷들) 특징

원본은 전체적으로 아래 인상을 준다.

- 배경은 크게 흐림 처리된 인물 이미지
- 중앙 상단에 프로필 원형 이미지
- 프로필 밑에 이름/상태
- **검은/반투명 사각 대사 박스** 1개 중심
- 대사 박스 안의 텍스트는 **한 글자씩 생성**됨
- 듣는 중/말하는 중 상태가 화면 중앙에 자연스럽게 섞임
- 응답 방식은 다음 중 하나:
  - `다음`
  - 선택지 2~3개
  - 직접 답하기 + 전송
- 통화 종료 시도 별도 카드/상태 제공
- 프로필 주변 링이 은은하게 살아 있고, 통화 중 느낌이 더 강함

### 원본의 본질

원본은 **전화 통화를 시네마틱한 인터랙티브 장면처럼 연출**한다.
즉,

```text
전화가 연결됨
-> 캐릭터의 첫 발화 카드가 한 글자씩 생성
-> 다 보이면 사용자 액션 가능
-> 선택지 또는 다음 또는 직접답하기
-> 사용자가 응답
-> 다음 캐릭터 발화 카드 생성
-> 반복
```

이게 핵심이다.

## 2. 이번 작업의 핵심 목표

### 최종 목표
현재 RN 통화 화면을 아래 기준으로 바꾼다.

1. **대사 표시 방식**을 “누적 transcript 중심”에서 “현재 턴 카드 중심”으로 수정
2. **사각 대사 박스 안 글자 타이핑 애니메이션** 구현
3. **프로필 사진 주변 통화 링/펄스** 구현
4. **연결 중/말하는 중/듣는 중/응답대기/종료 중** 상태 연출 강화
5. **한 턴이 끝나면 다음/선택지/직접답하기가 나타나는 원본 흐름** 구현
6. **직접답하기는 보조수단**, 선택지/다음은 주된 진행 수단으로 정리
7. **원본처럼 통화 장면 하나를 보고 있는 듯한 레이아웃**으로 재구성

## 4. 가장 중요한 구조 변경: 통화 UI 패러다임 수정

## 4-1. 현재 구조(문제)

```text
Hero
+ transcript list
+ 선택지/입력
+ 끊기
```

이 구조는 채팅 앱 느낌이 강하다.

## 4-2. 바꿔야 하는 구조(원본형)

```text
[상단 닫기/X 또는 뒤로가기]
[프로필 이미지 + 이름 + 상태]
[현재 발화 카드 1개]
[듣는 중 / 응답 대기 상태 라벨]
[다음 버튼 또는 선택지 또는 직접답하기]
[통화 끊기]
```

즉, **통화는 “현재 장면” 중심**으로 보여야 한다.

### 강한 권장
- 긴 transcript를 기본 UI에서 크게 누적하지 말 것.
- 필요하다면 내부 state로 transcript는 계속 저장하되, **화면에는 현재 turn 중심으로만 보여라.**
- 최근 1~2개의 발화 흔적만 남길 수는 있지만, 기본 연출은 단일 현재 카드가 더 중요하다.

## 5. 상태 머신 재설계

통화 화면은 반드시 아래 상태를 명시적으로 가져야 한다.

```ts
type CallPhase =
  | 'dialing'            // 전화 거는 중
  | 'ringing'            // 연결 대기 / 벨 울림
  | 'connected'          // 연결됨
  | 'character_typing'   // 캐릭터 발화 타이핑 중
  | 'awaiting_next'      // 다음 버튼 대기
  | 'awaiting_choice'    // 선택지 대기
  | 'awaiting_text'      // 직접답하기 입력 대기
  | 'user_sending'       // 사용자 발화 전송 중
  | 'listening'          // 듣는 중 상태
  | 'ending'             // 종료 중
  | 'ended';             // 종료 완료
```

추가로 현재 턴 타입도 분리해라.

```ts
type CallTurnUiMode =
  | 'monologue'      // 다음 버튼 하나로 넘김
  | 'choices'        // 선택지 2~3개
  | 'input'          // 직접답하기 중심
  | 'mixed';         // 선택지 + 직접답하기
```

## 6. 데이터 모델 권장안

```ts
type CallTurn = {
  id: string;
  speaker: 'character' | 'user' | 'system';
  text: string;
  displayText?: string;   // typewriter 진행 중 보여줄 텍스트
  phase?: 'pending' | 'typing' | 'done';
  modeAfter?: CallTurnUiMode;
  choices?: string[];
  canDirectReply?: boolean;
  createdAt: number;
};

// 통화 세션

type CallSession = {
  open: boolean;
  roomId: string;
  characterId: string;
  phase: CallPhase;
  turns: CallTurn[];              // 실제 컨텍스트용 누적 기록
  currentTurnId?: string;         // 현재 화면에 강조 렌더링할 턴
  visibleTurnIds?: string[];      // 최근 1~2개 정도만 표시하고 싶다면 사용
  inputMode: 'hidden' | 'collapsed' | 'expanded';
  draftText: string;
  durationSec: number;
  connectedAt?: number;
  callStartedAt?: number;
  ringPulse: boolean;
  lastAction?: 'next' | 'choice' | 'text' | 'hangup';
};
```

### 핵심
- `turns`는 내부 데이터용
- `currentTurnId`는 화면 연출용
- UI는 `currentTurnId` 중심으로 그린다.
- transcript를 다 그리는 방식이 아니라 **현재 턴 + 직전 턴 일부**만 보여라.

## 7. 프로필 영역(통화 링 포함) 작업지시

## 7-1. 프로필 레이아웃

구성:
- 상단 좌측: 뒤로가기(`<`) 또는 닫기(`×`)
- 중앙: 원형 프로필 사진
- 프로필 아래: 캐릭터 이름
- 이름 아래: 상태 문구 (`통화 중`, `통화 연결됨`, `듣는 중...`, `내 응답을 기다리는 중`, `통화 종료 중`)

### 권장 크기
- 프로필 이미지: 120~140dp
- 프로필 바깥 링 포함 컨테이너: 144~168dp
- 이름: 26~34sp, bold
- 상태문구: 13~16sp, medium

---

## 7-2. 프로필 주변 링 애니메이션 (중요)

원본 느낌을 살리기 위해 프로필 사진 주변에 **상태별 링 연출**을 넣는다.

### A. dialing / ringing
- 바깥 링이 2중 또는 3중으로 은은하게 pulse
- opacity 0.12 -> 0.26 -> 0.12
- scale 1.0 -> 1.08 -> 1.0
- duration 1200~1600ms
- loop

### B. connected / idle
- 고정 얇은 링 1개
- 아주 약한 breathing
- opacity 0.18 -> 0.24 -> 0.18
- duration 2200~3000ms

### C. character_typing / speaking
- 링이 좀 더 또렷하게 pulse
- scale 1.0 -> 1.05 -> 1.0
- opacity 0.22 -> 0.35 -> 0.18
- duration 1000~1300ms
- 말하는 중 느낌을 주기 위해 line color를 약간 더 밝게

### D. listening / awaiting_user
- 링은 고정
- 상태 문구만 `내 응답을 기다리는 중` 또는 `듣는 중...`
- pulse 강도는 줄인다.

### E. ending
- 링과 전체 opacity가 서서히 약해짐
- 400~700ms 동안 fade

### RN 구현 힌트
- `Animated.View` 2~3개로 outer ring 겹치기
- `transform: [{ scale }]`, `opacity`
- backgroundColor는 투명, borderColor만 사용

## 8. 대사 박스(Card) 연출 작업지시

## 8-1. 핵심 방향

원본과 가장 큰 차이는 **대사 카드**다.

현재 구현은 말풍선/채팅 버블 느낌이 아직 강한데, 원본은 **중앙에 크게 놓이는 반투명 사각 박스**가 중심이다.

### 카드 스타일 지시
- 위치: 프로필 영역 아래 중앙
- 최대 너비: 웹 640~760px, 모바일 가로 80~88%
- 최소 높이: 96~120dp
- 배경: `rgba(0,0,0,0.42 ~ 0.58)` 또는 아주 어두운 반투명
- border: 1px solid rgba(255,255,255,0.08)
- radius: 16~22dp
- padding: 22~28dp
- 텍스트: 흰색 계열
- lineHeight: 1.45~1.6
- 그림자/블러: 약하게

### 대사 카드 기본 원칙
- 기본적으로 **지금 말하는 카드 1개가 가장 중요**하다.
- 캐릭터 발화는 이 카드에 노출된다.
- 이전 발화 흔적을 남기고 싶으면 바로 위/아래에 아주 약하게 1개만 남겨라.
- 현재처럼 3~4개의 큰 버블이 계속 누적되는 방식은 지양.

---

## 8-2. 카드 안 텍스트 타이핑 애니메이션 (가장 중요)

### 요구사항
원본처럼 **사각 박스 안에 한글자씩 생성되는 느낌**을 구현한다.

### 규칙
1. 캐릭터 발화 시작 시 빈 카드 생성
2. 카드 내부 텍스트는 `displayText = ''`부터 시작
3. 1글자씩 append
4. 완료 전까지는 선택지/다음 버튼 비활성 또는 숨김
5. 타이핑 완료 후에야 다음 인터랙션 노출

### 속도 권장값
- 기본: 18~34ms / 글자
- 문장부호(`, . ? ! …`) 뒤: +90~160ms pause
- 줄바꿈 뒤: +120~220ms pause
- 긴 문장 중간 쉼: +50~90ms

### 매우 중요
- 한글 조합이 깨지지 않게 **문자 단위** 처리 시 주의
- JS에서 `Array.from(text)` 사용 권장

### pseudo-code

```ts
async function typeTurnText(turnId: string, fullText: string) {
  const chars = Array.from(fullText);
  let out = '';

  updateTurn(turnId, { displayText: '', phase: 'typing' });

  for (const ch of chars) {
    out += ch;
    updateTurn(turnId, { displayText: out, phase: 'typing' });

    let delay = 26;
    if (/[,.!?…]/.test(ch)) delay += 120;
    if (/\n/.test(ch)) delay += 160;

    await sleep(delay);
  }

  updateTurn(turnId, { displayText: fullText, phase: 'done' });
}
```

## 8-3. 한 마디마다 “다음 페이지”처럼 넘기는 구조

원본 느낌을 살리려면, 캐릭터의 긴 발화를 한 번에 너무 길게 보여주기보다 **문장 덩어리/턴 단위**로 잘라서 넘기는 게 중요하다.

### 권장 방식

#### A안 (가장 추천)
- LLM에서 `lines` 1~3개를 받음
- 각 `line`을 **하나의 카드 페이지**처럼 순차 노출
- 각 line이 다 타이핑되면:
  - `다음` 버튼 또는
  - 자동 350~700ms 후 다음 line 진행
- 마지막 line 후 선택지 등장

#### B안
- `lines`를 한 카드 안에 순차 append
- 하지만 카드 내부에서도 문단별 “다음 넘김” 체감이 있어야 함

### 추천 결론
**A안이 원본에 가장 가깝다.**
즉,

```text
1페이지: “오빠 목소리 들으니까 갑자기 울컥한다...”
[다음]
2페이지: “나 오늘 진짜 힘들었단 말이야.”
[다음]
3페이지: “오빠는 내 맘 다 알지?”
[선택지 등장]
```

현재 RN에서는 캐릭터 발화 1턴 안의 `lines[]`를 순차 page처럼 처리하라.

## 9. 상태 문구/보조 라벨 작업지시

원본에서 상태 문구는 꽤 중요하다.

### 상태 문구 규칙

| phase | 상태 문구 예시 |
|---|---|
| dialing | `연결 중...` |
| ringing | `연결 중...` 또는 `벨이 울리는 중...` |
| connected | `통화 연결됨` |
| character_typing | `정선이 말하는 중...` |
| listening | `듣는 중...` |
| awaiting_choice | `내 응답을 기다리는 중` |
| awaiting_next | `다음으로 진행 가능` 또는 숨김 |
| awaiting_text | `직접 답하기` 보조 문구 |
| ending | `통화 종료 중` |

### 연결 시간 pill
원본 느낌을 위해 `통화 연결됨 · 00:23` 같은 pill을 유지한다.

스타일:
- 배경: 반투명 어두운 pill
- 폰트: 13~15sp
- padding: 10~14dp / 16~18dp
- radius: 999dp

## 10. 사용자 응답 UI 작업지시

## 10-1. 선택지 기본형

원본은 선택지 버튼이 매우 중요하다.

### 규칙
- 캐릭터 턴이 끝난 뒤에만 선택지 노출
- 선택지는 2~3개 권장
- 세로 스택
- 버튼 크고 두껍게
- 원본처럼 흰색/밝은 배경 버튼

### 스타일
- 높이: 52~64dp
- radius: 12~16dp
- 텍스트: 16~20sp, bold
- 간격: 12~16dp
- 그림자 약간

### UX 규칙
- 선택지 누르면 즉시 버튼 숨김
- `listening` 또는 `user_sending` 상태로 전환
- 필요하면 선택한 문장은 잠깐 카드나 로그로 보여줄 수 있으나, 원본풍을 살리려면 **크게 노출하지 않아도 된다.**

---

## 10-2. `다음` 버튼

원본 스샷처럼 어떤 턴은 선택지 대신 `다음` 하나만 보여야 한다.

### 언제 사용?
- 캐릭터가 혼잣말/설명/상황 전달을 하고 사용자 의미 있는 분기를 아직 요구하지 않을 때
- 통화 연결 직후 인사/도입부
- 마지막 goodbye line 다음 종료 진행

### 스타일
- 선택지 버튼과 유사하되 단일 버튼
- 라벨: `다음`

### 동작
- 다음 line/page로 이동
- 또는 다음 턴 생성 트리거

---

## 10-3. 직접 답하기

현재 RN에서 직접 답하기가 너무 메인으로 보이면 원본과 어긋난다.

### 원칙
- 직접 답하기는 **보조 수단**이다.
- 기본은 선택지/다음 중심
- `직접 답하기`는 작은 텍스트 버튼 또는 입력칸 펼침 트리거로 처리

### 추천 UI
- 선택지 아래 `직접 답하기` 텍스트 버튼
- 누르면 입력칸 확장
- 입력칸 + `전송`

### 스타일
- 입력칸 폭: 화면의 60~75%
- 버튼: 소형 노랑/강조 버튼
- placeholder: `직접 답하기`

### 중요
`직접 답하기`가 항상 크게 열려 있지 않게 하라. 원본은 그때그때 필요한 순간만 등장하는 느낌에 더 가깝다.

---

## 11. 끊기 버튼 작업지시

### 유지 원칙
- 하단 중앙 또는 하단 고정
- 빨간색 계열 강조
- 터치 영역 큼

### 스타일
- 모바일: 가로 거의 전체, height 58~64dp
- 웹: 180~240px 정도 폭의 pill or rounded rect
- radius: 18~28dp
- 폰트: 18~22sp, bold

### 동작
- 누르면 바로 종료하지 말고
  - `ending` 상태
  - 원본처럼 마지막 대사 한 줄이 가능하면 생성
  - 이후 종료

## 12. 배경 연출 작업지시

원본은 흐린 인물 배경이 핵심 무드다.

### 규칙
- 캐릭터 대표 이미지 또는 프로필 이미지를 확대/크롭해서 전체 배경으로 깐다.
- 강한 blur + dark overlay 적용
- 프로필 이미지와 같은 인물임을 은근히 느끼게만 하면 됨.

### 스타일
- `ImageBackground`
- blurRadius 강하게
- 그 위에 `rgba(0,0,0,0.35~0.55)` overlay
- 카드/텍스트가 읽히도록 대비 유지

## 13. 실제 턴 렌더링 규칙

## 13-1. 캐릭터 턴

1. phase -> `character_typing`
2. 카드 fade in
3. 빈 카드에서 typewriter 시작
4. 텍스트 완료
5. 상태 문구 -> `내 응답을 기다리는 중` or `듣는 중...`
6. 선택지/다음/직접답하기 노출

## 13-2. 사용자 선택지 선택

1. 버튼 tap
2. 버튼 press animation
3. 버튼 숨김
4. phase -> `listening` 또는 `user_sending`
5. 원하면 작은 user confirm UI 잠깐 표시
6. 다음 캐릭터 turn 생성

## 13-3. 직접답하기 전송

1. 입력값 trim
2. 비어있으면 무시
3. phase -> `user_sending`
4. 입력칸 접힘 또는 비활성
5. 캐릭터 다음 턴 생성

## 13-4. 종료 턴

1. 사용자가 `끊기`
2. phase -> `ending`
3. 상태문구 -> `통화 종료 중`
4. 마지막 goodbye line typewriter
5. fade out
6. call log 저장

## 14. 프롬프트/턴 스키마 보강 지시

통화 UI가 원본처럼 되려면, 모델 출력도 원본형 UI에 맞아야 한다.

### 추천 스키마

```json
{
  "lines": [
    "오빠 목소리 들으니까 갑자기 울컥한다...",
    "나 오늘 진짜 힘들었단 말이야.",
    "오빠는 내 맘 다 알지?"
  ],
  "choices": [
    "다시 말해줘.",
    "괜찮아. 천천히 해.",
    "끊을게."
  ],
  "uiMode": "choices",
  "allowDirectReply": true
}
```

### 프롬프트 규칙
- `lines`는 1~3개 짧은 spoken lines
- 각 줄은 실제 전화 통화 말투처럼 짧고 자연스럽게
- narration 금지
- stage directions 금지
- speaker label 금지
- `uiMode`는 `next`, `choices`, `input`, `mixed` 중 하나
- 캐릭터는 line 하나를 한 카드 페이지처럼 사용한다고 가정하고 작성

### 종료 프롬프트

```json
{
  "line": "어, 알겠어. 조심히 들어가고... 나중에 다시 연락해."
}
```

## 18. 테스트 시나리오

수정 후 아래 시나리오가 반드시 성립해야 한다.

### 시나리오 1: 발신 직후
- 통화 버튼 탭
- `연결 중...`
- 프로필 링 pulse
- 연결되면 `통화 연결됨`
- 첫 line 카드가 빈 상태에서 한 글자씩 생성

### 시나리오 2: 다음 버튼 흐름
- 캐릭터 도입부 line 출력
- 완료 후 `다음` 버튼 등장
- `다음` 탭 시 다음 page/line

### 시나리오 3: 선택지 흐름
- 마지막 캐릭터 line 완료
- 선택지 2~3개 등장
- 선택지 누르면 숨겨지고 다음 캐릭터 카드 생성

### 시나리오 4: 직접답하기
- `직접 답하기` 탭
- 입력칸 펼침
- 전송 후 입력칸 접히고 캐릭터 응답 시작

### 시나리오 5: 듣는 중 상태
- 사용자 액션 후 캐릭터 응답 생성 직전
- `듣는 중...` 또는 `정선이 말하는 중...`가 자연스럽게 표시

### 시나리오 6: 종료
- `끊기` 탭
- `통화 종료 중`
- 마지막 한 줄 표시 가능
- fade out 후 종료

## 19. 바이브코딩툴에 넣을 핵심 지시문

아래 블록은 그대로 사용할 수 있다.

```text
SNSGod 통화 기능을 원본 스크린샷 느낌으로 복각해줘.

현재 RN 구현은 통화 transcript형 UI가 강한데, 원본은 “통화 장면형 / 카드형 UI”에 가깝다. 따라서 아래 요구사항에 맞춰 구조와 애니메이션을 수정해줘.

핵심 목표:
1. 통화 화면을 transcript 누적형이 아니라 현재 턴 카드 중심 구조로 바꾼다.
2. 캐릭터 대사는 중앙의 반투명 사각 카드 안에서 한 글자씩 typewriter 애니메이션으로 생성되게 한다.
3. 프로필 사진 주변에 통화 링/pulse 애니메이션을 넣는다.
4. 연결 중 / 통화 연결됨 / 말하는 중 / 듣는 중 / 내 응답을 기다리는 중 / 통화 종료 중 상태를 phase 기반으로 분리한다.
5. 캐릭터 발화는 1~3개의 짧은 spoken lines로 받고, 각 줄은 한 페이지처럼 순차 표시한다.
6. 어떤 턴은 `다음` 버튼으로 진행하고, 어떤 턴은 선택지 2~3개를 노출한다.
7. 직접 답하기는 기본이 아니라 보조 기능으로 처리하고, 필요 시에만 펼친다.
8. 끊기 시 종료 중 상태와 마지막 한마디를 처리할 수 있게 한다.
9. 배경은 캐릭터 이미지 기반 blur background + dark overlay 구조로 만든다.
10. 웹/모바일 모두 원본 느낌을 살리되, 모바일 우선 레이아웃을 기준으로 구성한다.

상태 머신:
- dialing
- ringing
- connected
- character_typing
- awaiting_next
- awaiting_choice
- awaiting_text
- user_sending
- listening
- ending
- ended

UI 구조:
- top bar (뒤로가기 또는 X)
- hero (프로필, 이름, 상태)
- status pill (통화 연결됨 · 00:23)
- current turn card (반투명 사각 박스)
- action area (다음 / 선택지 / 직접답하기)
- hangup button

대사 카드 규칙:
- 중앙의 반투명 dark card
- white text
- padding large
- radius 16~22
- max width 80~88% mobile, 640~760px web
- typewriter로 한 글자씩 생성
- 완료 전에는 다음 액션 비활성 또는 숨김

프로필 링 애니메이션:
- ringing: 2~3중 outer ring pulse
- connected: 약한 breathing
- speaking: 좀 더 또렷한 pulse
- listening: 고정 또는 약한 pulse
- ending: fade down

턴 진행 규칙:
- model returns lines[] + choices[] + uiMode + allowDirectReply
- lines는 1~3개 짧은 대사
- 각 line은 한 페이지처럼 순차 진행
- 마지막 line 후 uiMode에 따라 다음/선택지/입력창을 띄운다

직접답하기:
- 기본 숨김
- `직접 답하기` 텍스트 버튼 누를 때만 펼침
- 입력칸 + 전송 버튼

절대 유지/변경 제외:
- Grok 이미지 생성 API
- Vertex JSON API 기본 구조
- 카카오톡 테마
- 캐릭터 reference image 옵션
- 방별 추가 프롬프트 입력 기능

완료 기준:
- 통화 시작 시 연결 중 + 링 pulse가 보인다.
- 캐릭터 첫 대사가 카드 안에서 한 글자씩 생성된다.
- line/page 단위로 다음 또는 선택지 진행이 된다.
- 현재처럼 긴 transcript 느낌이 아니라, 원본처럼 “현재 장면” 중심으로 보인다.
- 직접답하기는 보조 기능으로 동작한다.
- 종료 시 종료 중 상태와 마지막 대사 처리가 가능하다.
```
