# SNSGod Oracle 메시지 서버 전환 및 모바일 구조 개선 작업계획서

- 작성일: 2026-07-10
- 대상 저장소: `wrem123-design/snsgod`
- 대상 앱: `mobile-rn` (`com.snsgod.rn`)
- 운영 범위: 개인 전용, 외부 배포 없음
- 서버 환경: Oracle Cloud Ubuntu 24.04 ARM64, 4 OCPU, 24GB RAM
- 기존 서버 기능: Grok 이미지 생성 API, Custom/Cerebras API 설정

## 1. 작업 목표

이 작업의 최종 목표는 세 가지다.

1. 앱을 완전히 종료해도 Oracle 서버가 1:1 답장, 1:1 선톡, 단톡방 답장과 선톡을 실제 시각에 생성하고 보관한다.
2. 현재 모바일 앱 전체 구조를 재검토하여 불필요한 백그라운드 자원 사용, 중복 로직, 지나치게 큰 파일, 저장 불안정 요소와 유지보수하기 어려운 부분을 정리한다.
3. 사용자가 복잡한 수치를 몰라도 캐릭터별·방별 행동을 쉽게 설정할 수 있고, 필요하면 고급 설정까지 세밀하게 조정할 수 있는 구조를 만든다.

완료 후 사용 경험은 다음과 같아야 한다.

- 앱을 몇 시간 또는 며칠 실행하지 않아도 서버에서 메시지가 생성되고 누적된다.
- 휴대폰이 온라인이면 실제 알림이 도착한다.
- 휴대폰이 오프라인이거나 알림이 지연돼도 앱을 열면 서버에 저장된 메시지를 원래 생성 시각으로 내려받는다.
- 앱을 연 직후 과거 시각을 붙여 새로 생성하는 방식이 사라진다.
- 서버 메시지는 중복 없이 한 번만 대화방에 들어가며, 재설치·서버 재시작·네트워크 재연결에도 정합성이 유지된다.

## 2. 현재 구조 재검토 결과

### 2.1 현재 메시지 실행 구조

- `App.tsx`가 앱 시작 후 5초 뒤 첫 자동화 작업을 실행하고, 이후 1분마다 자동화 틱을 실행한다.
- `AutomationKeepAliveService`가 포그라운드 서비스와 WakeLock으로 앱 프로세스 생존 가능성을 높인다.
- 앱이 강제 종료되면 자동화와 AI 생성이 계속될 수 없다.
- 1:1 답장은 `replyEngine.ts`, 단톡방 답장은 `GroupChatRoomScreen.tsx`, 선톡은 `automation.ts`에 분산돼 있다.
- 늦어진 답장을 앱 재실행 시 과거 시각으로 보정하는 catch-up 로직이 있으나, 실제 생성 시점은 앱을 다시 실행한 이후다.
- 현재 `notifications.ts`는 앱 내부 알림 목록이며 실제 원격 푸시 수신 구조가 아니다.

### 2.2 현재 저장 구조

- 전체 상태는 `SNSGodState`라는 큰 상태 객체에 집중돼 있다.
- SQLite, AsyncStorage 포인터, 파일 백업을 함께 사용하며 복구·검증 로직이 `persist.ts` 한 파일에 크게 집중돼 있다.
- 메시지는 SQLite 테이블로 분리돼 있지만 나머지 도메인 상태와 저장 책임이 강하게 결합돼 있다.
- 스키마 버전은 존재하지만 서버 동기화용 ID, 동기화 커서, 멱등성 키, 서버 생성 시각, 전송 상태 모델이 없다.
- 백업 내 API 키 제거 로직은 있으나 서버 동기화용 명시적 데이터 필터가 별도로 필요하다.

### 2.3 현재 유지보수 위험

- `SettingsScreen.tsx`, `blindDate.ts`, `persist.ts`, `api.ts`, 채팅 화면 등 여러 파일이 1,000줄 안팎 또는 그 이상이다.
- 화면 컴포넌트 안에 AI 호출, 상태 변경, 시간 지연, 저장 처리가 함께 들어간 부분이 있다.
- 1:1과 단톡방이 유사한 메시지 생성 문제를 서로 다른 방식으로 처리한다.
- 자동 테스트 파일이 확인되지 않았다.
- 현재 `npm run check`는 `bootstrapGlobals.ts`, `polyfills.ts`의 `performance` 타입 오류 2건으로 실패한다.
- 기본 상태, 백그라운드 서비스 등 일부 한국어 문자열에 인코딩 손상 흔적이 있다.
- Android Manifest에 광범위한 저장소 권한, 배터리 최적화 제외, WakeLock, 포그라운드 서비스, cleartext 통신 허용 등이 들어 있어 서버 전환 후 필요성을 다시 판단해야 한다.

## 3. 목표 아키텍처

```text
[휴대폰 앱]
  사용자 메시지 작성
        |
        v
[Oracle Message Service]
  인증 / 동기화 / 예약 판단 / 대화 맥락 관리
        |
        +--> [Custom/Cerebras API] 답장 또는 선톡 문장 생성
        |
        +--> [SQLite DB] 생성 메시지와 전송 상태 저장
        |
        +--> [Firebase FCM] 새 메시지 알림 전송
        |
        v
[휴대폰 앱]
  푸시 수신 또는 앱 실행 시 증분 동기화
```

역할 분담 원칙은 다음과 같다.

### Oracle 서버가 담당할 작업

- 1:1 사용자 메시지에 대한 답장 예약과 생성
- 1:1 캐릭터 선톡 시각과 확률 계산
- 단톡방 답장 발신자·발신 순서·말풍선 수 계산
- 단톡방 선톡과 캐릭터 간 대화 생성
- 최근 대화 맥락, 방 설정, 캐릭터 설정 보관
- 생성된 메시지 영구 저장
- FCM 알림 발송
- 재시도, 중복 방지, 실패 기록, 상태 확인

### 휴대폰이 담당할 작업

- 채팅 화면 표시와 입력
- 메시지 로컬 캐시
- 읽음 상태와 UI 상태 반영
- 서버에 사용자 메시지 업로드
- 서버에서 새 메시지 증분 수신
- 푸시 알림 수신과 해당 대화방 열기
- 이미지·스티커 등 큰 개인 미디어의 로컬 관리

### Custom/Cerebras API가 담당할 작업

- 서버가 전달한 프롬프트를 바탕으로 실제 텍스트 생성
- 1:1 또는 단톡방용 구조화된 JSON 응답 생성

Oracle 서버는 대형 AI 모델을 직접 실행하지 않고 예약·맥락·저장·중계 역할을 하므로 개인 사용 기준 트래픽과 CPU 사용량은 낮게 유지한다.

## 4. 서버 구성 계획

### 4.1 배포 원칙

- 기존 Grok 이미지 서버와 별도 디렉터리·프로세스·환경변수·데이터 볼륨으로 분리한다.
- 기존 이미지 서버의 포트와 설정은 변경 전에 읽기 전용으로 조사한다.
- ARM64에서 동작하는 런타임과 컨테이너만 사용한다.
- 외부 통신은 HTTPS만 허용하고, API 키와 Firebase 서비스 계정은 서버 환경변수 또는 제한된 비밀 파일로만 관리한다.
- 개인용이더라도 무인증 공개 API로 만들지 않는다.

### 4.2 권장 서버 모듈

- `api`: 기기 등록, 초기 동기화, 메시지 송수신, 설정 동기화
- `scheduler`: 예약된 답장·선톡 작업 탐색과 실행
- `generation`: Custom/Cerebras 요청, 응답 검증, 재시도
- `push`: Firebase FCM 발송
- `repository`: SQLite 트랜잭션과 마이그레이션
- `auth`: 단일 사용자용 기기 비밀 토큰 인증
- `observability`: 상태 점검, 구조화 로그, 실패 작업 조회

### 4.3 권장 DB 테이블

- `devices`: 기기 ID, 인증 토큰 해시, FCM 토큰, 마지막 접속 시각
- `characters`: 서버 생성에 필요한 캐릭터 설정과 버전
- `rooms`: 1:1·단톡방 정보, 활성화 여부, 마지막 활동 시각
- `room_participants`: 단톡방 참여 캐릭터와 발화 옵션
- `messages`: 서버/클라이언트 메시지 ID, 방 ID, 발신자, 내용, 실제 생성 시각
- `message_jobs`: 답장·선톡 예약 시각, 상태, 재시도 횟수, 실패 사유
- `room_contexts`: 요약, 최근 메시지 커서, 메모리 버전
- `automation_settings`: 전역·방별·캐릭터별 자동화 설정
- `sync_cursors`: 기기별 마지막 동기화 위치
- `push_outbox`: 푸시 발송 대기 및 결과
- `dead_letters`: 반복 실패 작업 격리

### 4.4 최소 API 계약

- `POST /v1/device/register`: 기기와 FCM 토큰 등록
- `POST /v1/sync/bootstrap`: 캐릭터·방·최근 메시지 최초 동기화
- `POST /v1/messages`: 사용자 메시지 업로드 및 답장 예약
- `GET /v1/sync/changes?cursor=`: 서버 생성 메시지 증분 수신
- `POST /v1/messages/ack`: 수신·읽음 확인
- `PUT /v1/settings/automation`: 자동화 설정 갱신
- `POST /v1/push/test`: 테스트 알림
- `GET /health`: 서버·DB·스케줄러 상태 확인

모든 쓰기 API는 `requestId` 또는 `clientMessageId`를 사용해 재전송돼도 중복 생성되지 않도록 한다.

### 4.5 1:1 메시지 규칙

- 사용자가 메시지를 보내면 서버가 캐릭터별 읽기 지연 범위 안에서 `dueAt`을 생성한다.
- 예정 시각이 되면 서버가 최신 방 맥락으로 답장을 생성한다.
- 생성된 각 말풍선은 별도 메시지 ID와 생성 시각을 갖는다.
- 선톡은 최소 간격, 발생 확률, 조용한 시간, 사용자의 마지막 활동, 연속 선톡 제한을 함께 적용한다.
- 사용자가 답하지 않을 때 이어 말하기 횟수를 캐릭터별로 제한한다.

### 4.6 단톡방 메시지 규칙

- 사용자 메시지에 반응할 후보 캐릭터와 발화 우선순위를 서버가 선택한다.
- 동일 캐릭터 연속 발화 방지, 최근 발화량 균형, 참여 제외 캐릭터, 최대 말풍선 수를 적용한다.
- 단톡 선톡은 한 명만 말하기, 두 명 대화, 전체 대화 등의 모드를 지원한다.
- 여러 말풍선의 순서와 시각을 서버에서 확정하고 원자적으로 저장한다.
- 일부 생성에 실패해도 이미 저장된 메시지를 중복 생성하지 않는다.

### 4.7 푸시 정책

- FCM은 메시지 원본 저장소가 아니라 새 메시지 알림 신호로만 사용한다.
- 메시지 원문은 항상 서버 DB에 먼저 저장한 뒤 푸시를 보낸다.
- 알림이 누락되거나 지연돼도 앱 실행 시 증분 동기화로 복구한다.
- 방 ID와 메시지 ID를 데이터에 포함해 알림을 누르면 정확한 대화방을 연다.
- 알림 내용 숨기기, 이름만 표시, 전문 표시를 사용자가 선택할 수 있게 한다.

## 5. 모바일 구조 개선 계획

### 5.1 우선 안정화

- 현재 TypeScript 오류 2건을 수정해 `npm run check`를 기준선으로 통과시킨다.
- 손상된 한국어 문자열을 전수 검색해 UTF-8로 복구한다.
- 현재 기기 상태와 전체 백업을 보존한 뒤 스키마 마이그레이션을 시작한다.
- 핵심 로직 테스트 기반을 추가한다.

### 5.2 계층 분리

현재의 큰 `SNSGodState` 중심 구조를 한 번에 폐기하지 않고 다음 계층으로 점진 분리한다.

- `domain`: Message, Room, Character, AutomationPolicy 등 순수 타입과 규칙
- `application`: 답장 요청, 선톡 예약, 읽음 처리, 동기화 유스케이스
- `infrastructure`: 로컬 SQLite, 서버 API, FCM, AI 제공자 어댑터
- `presentation`: 화면, 입력, 설정, 로딩·오류 표시

화면에서 직접 AI를 호출하거나 장시간 타이머를 실행하는 코드는 application 계층으로 이동한다.

### 5.3 메시지 엔진 통합

- 1:1 답장, 단톡방 답장, 자동 선톡이 공통 `MessageJob` 모델을 사용하도록 통합한다.
- 로컬 엔진과 서버 엔진이 같은 입력·출력 계약을 사용하게 한다.
- 서버 모드에서는 화면이 AI를 직접 호출하지 않고 메시지 작업을 서버에 요청한다.
- 서버 생성 메시지는 `serverMessageId`, `serverCreatedAt`, `syncStatus`를 갖는다.
- 중복 푸시·중복 동기화·재시작에도 동일 메시지가 한 번만 저장되게 한다.

### 5.4 저장소 단순화

- UI 전체 상태 저장과 메시지 저장 책임을 분리한다.
- 메시지·방·동기화 커서·전송 대기열을 명시적 SQLite 테이블로 관리한다.
- AsyncStorage는 가벼운 UI 설정과 DB 포인터 용도로 제한한다.
- 백업은 사용자 데이터, 미디어, 설정을 구분하고 API 키·서버 인증 토큰은 기본적으로 제외한다.
- 저장 실패와 복구 경로를 테스트 가능한 Repository 인터페이스로 감싼다.

### 5.5 네트워크·오프라인 처리

- 사용자가 보낸 메시지는 로컬에 즉시 표시하고 `전송 중/완료/실패` 상태를 표시한다.
- 네트워크가 없으면 outbox에 저장한 뒤 자동 재전송한다.
- 앱 시작, 포그라운드 복귀, 푸시 탭 시 증분 동기화를 실행한다.
- 같은 메시지를 여러 경로로 받아도 ID 기준으로 합친다.
- 서버 시각과 휴대폰 시각 차이를 보정한다.

### 5.6 Android 네이티브 정리

서버 전환이 검증되기 전까지 기존 포그라운드 서비스를 기능 플래그 뒤에 유지한다. 전환 후에는 다음을 재검토한다.

- WakeLock과 배터리 최적화 제외 요청 제거 가능 여부
- `FOREGROUND_SERVICE_DATA_SYNC`와 지속 알림 제거 가능 여부
- 과거 저장소 권한 제거 가능 여부
- `usesCleartextTraffic=true` 제거와 HTTPS 강제
- Termux 권한·모듈이 Grok 로컬 모드에 여전히 필요한지 분리
- FCM 수신 서비스, 알림 채널, 딥링크 추가

## 6. 사용자 커스터마이징 개선 계획

### 6.1 설정 구조 원칙

- 기본 설정은 결과 중심의 쉬운 표현을 사용한다.
- 복잡한 숫자는 고급 설정 안으로 이동한다.
- 전역 기본값 → 캐릭터별 덮어쓰기 → 방별 덮어쓰기 순서로 적용한다.
- 각 설정은 실제 효과를 바로 이해할 수 있는 설명과 예시를 제공한다.
- 설정 변경 전후를 미리 시험할 수 있게 한다.

### 6.2 쉬운 모드

자동화 화면에 다음 프리셋을 제공한다.

- `조용함`: 답장과 선톡이 드물고 야간 활동이 적음
- `자연스러움`: 일반적인 생활 리듬과 중간 빈도
- `자주 대화`: 답장이 빠르고 선톡이 비교적 잦음
- `밤에 활발`: 저녁·심야 비중 증가
- `직접 설정`: 모든 고급 옵션 사용

사용자는 우선 다음 항목만 보게 한다.

- 서버 자동 메시지 사용 여부
- 답장 속도: 빠름 / 보통 / 느림
- 선톡 빈도: 거의 없음 / 가끔 / 자주
- 활동 시간대
- 답이 없어도 이어 말할지
- 1:1·단톡방 각각 사용 여부
- 알림 내용 표시 방식

### 6.3 고급 모드

- 답장 최소·최대 지연
- 말풍선 최소·최대 간격

- 최소 재선톡 간격
- 사용자 무응답 시 최대 연속 메시지 수
- 조용한 시간과 요일별 활동 시간
- 캐릭터별 시간대·생활 리듬
- 단톡방 최대 응답 캐릭터 수
- 단톡방 한 번에 생성할 최대 말풍선 수
- 단톡방 발신자 균형과 연속 발화 제한
- 서버에 보낼 최근 메시지 수와 요약 길이
- 모델, 온도, 최대 토큰, 실패 재시도 횟수
- 이미지 자동 생성 허용 여부와 빈도

### 6.4 방별·캐릭터별 제어

- 각 1:1 방에 `서버 답장`, `서버 선톡`, `알림` 개별 토글을 둔다.
- 각 단톡방에 참여 캐릭터, 선톡 모드, 최대 발화자 수, 대화 활발도 옵션을 둔다.
- 캐릭터 설정에 말투·리액션·답장 속도·선톡 성향 프리셋을 유지하되 실제 서버 정책과 같은 모델을 사용한다.
- 전역 설정과 다를 경우 화면에 `개별 설정 사용 중`을 명확히 표시한다.

## 7. 보안과 개인정보

- SSH 개인키, Cerebras/Custom API 키, Grok 키, Firebase 서비스 계정은 Git과 앱 백업에 포함하지 않는다.
- 휴대폰과 서버 간 통신은 HTTPS와 단일 사용자 기기 토큰으로 인증한다.
- 기기 토큰 원문은 서버 DB에 저장하지 않고 해시로 비교한다.
- 로그에 메시지 전문과 API 키를 기본 기록하지 않는다.
- 서버 동기화 시 앱의 전체 백업 JSON을 그대로 보내지 않고 필요한 필드만 명시적으로 선택한다.
- Vertex 서비스 계정이나 휴대폰에 저장된 기존 API 키는 서버 동기화 대상에서 제외한다.
- SQLite DB와 환경변수 파일의 OS 권한을 제한하고 정기 백업한다.

## 11. 롤아웃과 롤백 전략

1. `local` 모드: 기존 앱 방식만 사용
2. `shadow` 모드: 서버가 작업을 계산·생성하지만 앱에는 전달하지 않고 결과만 비교
3. `hybrid` 모드: 1:1 답장부터 서버 사용, 나머지는 로컬 유지
4. `server` 모드: 1:1·단톡 답장과 선톡을 서버가 전담
5. 안정화 후 기존 포그라운드 자동화 제거

각 단계는 설정에서 이전 모드로 되돌릴 수 있어야 한다. 서버 장애 시 로컬 채팅 열람과 사용자 메시지 작성은 유지하고, 전송 대기 메시지는 서버 복구 후 재전송한다.

## 12. 최종 완료 기준

- 앱이 완전히 종료된 동안 서버에서 1:1·단톡 메시지가 실제로 생성·저장된다.
- 앱을 열면 이미 생성된 메시지가 원래 시각과 읽지 않음 상태로 나타난다.
- FCM 알림이 가능한 경우 실제 시스템 알림이 표시된다.
- 알림 누락·서버 재시작·앱 재설치·네트워크 재연결에도 메시지가 중복되거나 사라지지 않는다.
- 사용자가 쉬운 프리셋 또는 고급 설정으로 답장 속도, 선톡 빈도, 활동 시간, 단톡 참여를 조절할 수 있다.
- TypeScript 검사와 핵심 테스트가 통과한다.
- 기존 Grok 이미지 생성 서버가 계속 정상 동작한다.
- WakeLock과 포그라운드 자동화에 의존하지 않는다.

## 13. 작업 시작 전 추가로 필요한 정보

- 기존 Grok 서버의 실행 방식: Docker Compose, systemd, PM2 중 무엇인지
- 기존 서버 소스·설정 디렉터리와 사용 중인 포트
- 현재 도메인·리버스 프록시·HTTPS 구성
- Oracle 보안 목록과 Ubuntu 방화벽의 80/443 상태
- Firebase 프로젝트 생성 여부
- SSH 개인키가 로컬에 준비돼 있는지 여부

비밀번호, SSH 개인키, API 키, Firebase 서비스 계정 JSON은 채팅이나 Git에 올리지 않는다. 실제 배포 시 서버 내부의 제한된 비밀 파일 또는 환경변수로만 설정한다.
