# 메시지 원문 보존 정책

SNSGod는 로컬 1인 사용 앱이므로 대화 원문 보존과 모델 입력 제한을 서로 다른 경계로 취급합니다.

## 영구 저장 경계

- `state.messages[roomId]`는 사용자가 명시적으로 메시지·방·캐릭터를 삭제하지 않는 한 전체 원문을 유지합니다.
- 저장 정규화는 방별 배열의 형식만 보정하고 개수로 자르지 않습니다.
- SQLite의 방별 메시지 저장과 파일 백업, 전체 ZIP 백업은 같은 전체 원문을 사용합니다.
- SQLite의 상태·메타데이터·방별 메시지는 하나의 transaction으로 저장합니다. 중간 쓰기가 실패하면 전부 rollback되어 서로 다른 시점의 데이터가 섞이지 않습니다.
- 일반적인 새 메시지 저장은 기존 immutable prefix를 확인한 뒤 SQLite에 새 행만 추가합니다. 과거 메시지 편집·삭제·미디어 URI 교체처럼 prefix가 달라진 경우에만 해당 방을 transaction 안에서 전체 교체합니다.
- 오래된 메시지의 사진도 state media reference graph에 남으므로 재시작과 백업 회전 뒤 reachability GC에서 보호됩니다.
- 120개를 넘었다는 이유로 개인톡, 단톡, 자동 단톡, 만남 이벤트 메시지를 삭제하지 않습니다.

## 모델 context 경계

- 답장 생성에는 저장 원문 전체가 아니라 `selectPromptContext`가 반환한 최근 window만 전달합니다.
- 기본값은 24개이며 가져온 설정이 비정상이거나 지나치게 커도 1~80개 범위로 제한합니다.
- 개인톡과 단톡은 같은 정책을 사용합니다. SNS, 전화, 자동화 등 별도 기능은 각 기능의 더 작은 전용 context를 계속 사용합니다.
- context 선택은 복사본을 만들 뿐 저장 배열을 변경하지 않습니다.

## 화면 렌더링 경계

- 채팅 화면은 전체 배열을 탐색할 수 있게 유지하되 React Native `FlatList`로 가상화합니다.
- 최초 24개, 배치당 16개, window 10 단위로 렌더링해 오래된 원문 수가 늘어도 모든 말풍선을 한 번에 마운트하지 않습니다.

## 성능과 회귀 검증

`mobile-rn/test/message-retention.test.mjs`는 다음을 확인합니다.

- 개인톡·단톡의 120/121 경계 보존
- 301개 단톡 원문의 JSON 재시작 round-trip
- 오래된 첫 메시지의 미디어 reachability
- 50,001개 원문의 append 계획, JSON round-trip 정규화, 24개 context 선택이 1초 안에 완료
- SQLite가 일반 append는 증분 쓰기, 편집·삭제는 전체 교체로 계획함
- 실제 SQLite 재시작과 강제 실패를 통해 121개 이상 원문·초기 미디어 보존 및 상태/메시지 원자적 rollback을 확인함
- 저장 경로에 silent trim이 다시 들어오지 않고 화면 가상화 설정이 유지됨
